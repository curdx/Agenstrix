<objective>
Harden the SQLite durability story locked in by Plan 01. Plan 01 ships a minimal `initDb()` per RESEARCH.md §Pattern 5; this plan extracts backup logic into a dedicated module, adds rigorous unit tests covering every PRAGMA invariant, fixes the WAL-checkpoint blocking-reader bug (RESEARCH.md Pitfall 7 — must be `PASSIVE` not `TRUNCATE`), and verifies the replay-ordering correctness that the WeChat-style history depends on.

If WAL grows unbounded, the user's `~/.agenstrix/store.db` becomes a multi-GB file within days. If a migration runs without backup, a Drizzle bug can drop a column irrecoverably. This plan locks these invariants behind tests so Plans 04, 05, 06 can refactor freely without regressing them.

Output: `src-bun/db/` is now defensively wired and test-covered. No user-facing change — the value is preventing data loss.

| Migration runner → schema | A buggy migration could DROP a column with PTY history; pre-migrate backup is the recovery path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-01-03-04 | Tampering | Concurrent appends produce duplicate `seq` values | mitigate | `appendAtomic` uses `db.transaction` with `SELECT MAX(seq)+1` + INSERT in one critical section |
